  <behavior>
    <!-- processor.test.ts behaviors — write tests FIRST, then implement -->

    processQueueBatch():
    - When there are no pending jobs, returns without error and does not call processJob
    - When a pending job has nextRetryAt <= now, it is marked 'processing' before processJob is called
    - When processJob resolves successfully, markJobDone() is called and the job ends with status='done'
    - When processJob rejects on attempt 0, markJobFailed() is called; job is re-queued as 'pending' with nextRetryAt in the future (backoff attempt 0 = 1 min delay)
    - When processJob rejects on attempt 2 (3rd failure, >= MAX_ATTEMPTS), markJobFailed() is called; job ends with status='failed'
    - A job with status='processing' and createdAt older than STALE_THRESHOLD_MS (2 minutes) is reset to 'pending' on the next processQueueBatch() call
    - A job with nextRetryAt > now is NOT picked up by dequeueNextBatch (correct backoff enforcement)
    - processQueueBatch() processes at most BATCH_SIZE (3) jobs per call, even if more are pending

    Additional invariants:
    - No test imports the production 'db' singleton — all tests inject via createTestDb()
    - processJob is a no-op stub that resolves in Phase 2 — test it as a spy that can be made to throw
  </behavior>

**src/background/queue/processor.ts**

Constants at the top of the file:
```typescript
const BATCH_SIZE = 3
const MAX_ATTEMPTS = 3
const STALE_THRESHOLD_MS = 2 * 60 * 1000  // 2 minutes
```

Backoff formula (base-5 exponential, 0-indexed attempt count):
```typescript
function backoffDelayMs(attemptIndex: number): number {
  return Math.pow(5, attemptIndex) * 60 * 1000
  // attemptIndex 0 → 60_000 ms (1 min)
  // attemptIndex 1 → 300_000 ms (5 min)
  // attemptIndex 2 → 1_500_000 ms (25 min)
}
```

Export `processQueueBatch(dbInstance?: BookmarkBrainDB): Promise<void>`. The optional `dbInstance` param follows the same pattern as Plan 01's CRUD functions — defaults to the production `db` singleton, allows test injection.

The function must:
1. **Stale job recovery pass (runs first):** Query for jobs with `status='processing'` AND `createdAt <= Date.now() - STALE_THRESHOLD_MS`. For each stale job, call `db.processingQueue.update(job.id, { status: 'pending', nextRetryAt: Date.now() })`. This handles mid-batch service worker termination without leaving jobs permanently stuck.

   Use a plain table filter for the stale pass (there is no compound index on `[status+createdAt]` for processingQueue — that index is on `bookmarks`). Full table scan is acceptable here because stale jobs are rare and the queue is small.

   Query: `(dbInstance ?? db).processingQueue.where('status').equals('processing').filter(j => j.createdAt <= Date.now() - STALE_THRESHOLD_MS).toArray()`

2. **Batch fetch:** Call `dequeueNextBatch(BATCH_SIZE, Date.now(), dbInstance)` — this uses the compound index and is fast.

3. **Process each job sequentially** (not concurrently — avoids race conditions in a single service worker):
   - Update job to `status: 'processing'` before starting
   - Call `await processJob(job)` (stub — see below)
   - On success: call `markJobDone(job.id, job.bookmarkId, dbInstance)`
   - On error: call `markJobFailed(job, errorMessage, dbInstance)` — `markJobFailed` handles the backoff/failed distinction internally

Export `processJob` as a named export so tests can spy on it. In Phase 2 it is a stub:
```typescript
export async function processJob(_job: ProcessingJob): Promise<void> {
  // Phase 5 will implement: extract page content, call AI API, write summary/tags back
}
```

Important: import `dequeueNextBatch`, `markJobDone`, `markJobFailed` from `'@/shared/db'`. Import `db` and `BookmarkBrainDB` from `'@/shared/db'` as well. Import `ProcessingJob` type from `'@/shared/types/db'`.
