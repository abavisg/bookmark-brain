# Project Research Summary

**Project:** Bookmark Brain — AI-powered Chrome Extension
**Domain:** Browser extension / local-first AI knowledge management
**Researched:** 2026-03-06
**Confidence:** HIGH

## Executive Summary

Bookmark Brain is a Chrome Manifest V3 extension that transforms passive bookmark saving into an active knowledge retrieval system. The architecture is fundamentally event-driven: a minimal popup triggers an async AI pipeline running in a background service worker, all data lives in IndexedDB via Dexie.js, and a separate full-page dashboard serves the primary browse and search experience. Research confirms that every major stack choice (MV3, Dexie, Vercel AI SDK, Readability.js) is well-established with clear community patterns — the technology risk is low. The risk is entirely in implementation discipline.

The MV3 service worker's ephemeral lifecycle is the single biggest threat to product stability. Chrome terminates workers after ~30 seconds of inactivity, and an LLM summarization pipeline can exceed this. The mitigation is non-negotiable from day one: persist a processing queue to storage and drive it through `chrome.alarms`, not a continuous async chain. Every other pitfall — content extraction edge cases, tag inconsistency, search zero-results — is solvable with well-understood techniques, but the queue architecture must be foundational. It cannot be retrofitted.

The recommended build order is infrastructure-first: scaffold the project, establish the data model with versioned schema, build the service worker message bus and alarm-driven queue, then wire in content extraction and the AI pipeline before touching any UI. Natural language search and AI-synthesized answers (the headline features) depend on a stable, well-tested content pipeline. Retroactive import shares that same pipeline code path, making it a late-phase feature that costs little to add once the core is solid.

## Key Findings

### Recommended Stack

The stack is built around the Chrome MV3 platform constraints with no custom backend. Vite with `@crxjs/vite-plugin` handles the multi-entry-point build (popup, dashboard, service worker, content script as four separate bundles). TypeScript is mandatory given the complexity of the bookmark + embedding + entity data model. Dexie.js is the only viable local storage choice — `chrome.storage.local` has no query API and a 10MB default cap.

For AI, the Vercel AI SDK (`ai` v4) provides the provider-agnostic interface the project requires. Users bring their own API key; the default provider is OpenAI with Anthropic as an immediately available alternative. Natural language search in v1 is LLM query-expansion over full-text IndexedDB queries — not embeddings. Embeddings are explicitly a v2 upgrade once user corpora exceed ~50K bookmarks.

**Core technologies:**
- **Chrome MV3 + Vite + @crxjs/vite-plugin**: Extension platform and build — the only viable modern Chrome extension build setup
- **TypeScript 5.x + React 18.x**: UI and type safety — complex data model demands TypeScript; React works well with crxjs and shadcn/ui
- **Dexie.js 4.x + dexie-react-hooks**: Local storage ORM — `useLiveQuery()` enables reactive UI without a state sync layer; handles schema migrations
- **Vercel AI SDK 4.x**: Provider-agnostic LLM abstraction — unified interface over OpenAI/Anthropic/Gemini; includes streaming, structured output, retry logic
- **@mozilla/readability 0.5.x**: Page content extraction — de facto standard (Firefox Reader View); handles ads/nav/boilerplate automatically
- **Tailwind CSS 4.x + shadcn/ui**: UI components — zero runtime, shared tokens across popup and dashboard
- **Zustand 4.x**: Ephemeral UI state — 1KB, no boilerplate; Dexie handles persistent state, Zustand handles UI state only
- **Vitest 2.x**: Testing — native TypeScript support, same Vite config

**Skip:** LangChain.js (over-abstracted, heavy bundle), SQLite WASM (unnecessary at v1 scale), vector databases (overkill until 50K+ bookmarks).

The full feature breakdown is in `FEATURES.md`. Summary for roadmap decisions:

**Must have (table stakes for v1):**

- Bookmark card: title, URL, date, favicon

- Tag-based organization (manual + auto-generated by AI)
- List/grid browse view on dashboard
- Delete/archive with undo

- Keyboard shortcut to save
- Filter by date range

**AI table stakes (defines the product category):**
- Automatic page summarization — 2-3 sentences, runs async post-save
- Auto-tagging from content analysis
- Natural language search in popup and dashboard
- Graceful degradation when AI fails (no key, paywall, SPA)

**Differentiators (competitive edge):**
- AI-synthesized answers with source bookmark cards (RAG over saved content)
- Entity/concept extraction (silent in v1, unlocks v2 knowledge graph)

- Export to Markdown/JSON

**Defer to v2+:**
- Cloud sync / multi-device
- User accounts
- Knowledge graph visualization
- Firefox/Safari support
- Embedding-based semantic search (upgrade from LLM query expansion)

### Architecture Approach

The architecture has seven distinct components: service worker (orchestrator), content script (DOM extractor), popup (quick-save + quick-search), dashboard (full library), Dexie storage layer (single source of truth), AI provider abstraction, and a small settings store in `chrome.storage.local`. The key design insight is that every component talks to the service worker via typed messages — nothing calls LLM APIs directly except the service worker. This is not optional: it is required by MV3 CORS rules and by the need to protect the API key.

**Major components:**
1. **Service Worker** — Central orchestrator. All LLM calls, queue management, and storage writes go through here. Must be designed as a stateless resumable process (terminated at any time).
2. **Storage Layer (Dexie + IndexedDB)** — Three tables: `bookmarks` (permanent metadata), `pageContent` (evictable raw text), `processingQueue` (resilient queue state). Versioned schema from day one.
3. **AI Provider Abstraction** — Thin interface over Vercel AI SDK. Three operations: `summarize()`, `search()`, `expandQuery()`. Provider swapped via settings.
4. **Content Script** — Readability.js extraction only. Sends result to service worker. No LLM calls, no API key access.
5. **Popup** — Minimal. Save button + quick search + last 5 bookmarks. Fast load (<200ms). State persisted to `chrome.storage.session`.
6. **Dashboard** — Standalone React app. Full library browse, NL search with synthesized answers, import flow, settings.
7. **Processing Queue + chrome.alarms** — Alarm-driven batch processor. The mechanism that survives service worker termination.

**Typed message protocol** connects all components — defined once in `shared/messages/`, used everywhere.

**Build structure:** `src/background/`, `src/content/`, `src/popup/`, `src/dashboard/`, `src/shared/` — four separate Vite bundles.

### Critical Pitfalls

The full pitfall list (15 pitfalls, severity-ranked) is in `PITFALLS.md`. The non-negotiable ones:

1. **Service worker terminated mid-AI-pipeline** — Use `chrome.alarms` as the batch scheduler from day one. Persist a processing queue to `chrome.storage.local` immediately on bookmark save. Design the pipeline as a resumable state machine with checkpoints. Never assume an async chain completes. Test with DevTools panel closed.

2. **Content extraction failures** — Real pages are auth-gated, JS-rendered SPAs, paywalls, PDFs, `chrome://` pages. Use Readability.js as primary extractor with a graceful fallback to `title + meta description`. Token-limit content before sending to LLM. Never block a save on extraction success.

3. **API key in content script context** — API key lives in the service worker only. Popup and content scripts route all LLM calls through `chrome.runtime.sendMessage`. Enforce this at the architecture level, not via convention.

4. **IndexedDB quota exhaustion** — Silent `QuotaExceededError` causes data loss. Always wrap writes in try/catch. Separate evictable `pageContent` from permanent `bookmarks` metadata. Call `navigator.storage.persist()` on startup. Warn at 70% quota.

5. **Retroactive import API cost spike** — 500 imported bookmarks = 500 LLM calls. Show cost estimate before import. Throttle to 3-5 concurrent. Require idempotency checks. Support pause/resume.

**Moderate pitfalls to plan for:**
- SPA injection timing (debounced MutationObserver)
- Tag inconsistency from LLM (normalize on write, constrain output format)
- Summarization blocking the save UX (optimistic save, async-only processing)
- Over-engineering search with vectors too early (ship full-text first)

## Implications for Roadmap

Research strongly suggests a 5-phase build sequence driven by the dependency chain: infrastructure enables AI pipeline enables search enables import.

### Phase 1: Foundation — Extension Infrastructure + Data Model

**Rationale:** Everything else is blocked until the build pipeline, MV3 manifest, storage schema, and message-passing architecture are in place. The service worker queue mechanism must be built here — retrofitting it later breaks everything.

**Delivers:** Working Chrome extension skeleton that can save a bookmark (no AI yet), store it in IndexedDB, and display it in a minimal popup. The alarm-driven processing queue exists and is exercised.

**Addresses (features):** Bookmark save gesture, visual save confirmation, bookmark card (title/URL/date), keyboard shortcut, basic list view.

**Avoids:** P1 (service worker termination), P3 (API key security), P4 (quota exhaustion), P8 (popup vs. dashboard conflict), P9 (content scripts calling LLM APIs), P10 (save blocked on AI), P12 (schema migration).

**Research flag:** None — standard patterns, well-documented.

---

### Phase 2: AI Processing Pipeline — Content Extraction + Summarization + Tagging

**Rationale:** The AI pipeline is the core product value. It must be stable before search can be built on top of it. Content extraction failures define the reliability of the entire system.

**Delivers:** Every saved bookmark gets a 2-3 sentence summary, auto-tags, and extracted entities — asynchronously after save, with graceful fallback if extraction or LLM fails.

**Addresses (features):** Automatic summarization, auto-tagging, entity extraction, graceful degradation, summary displayed on bookmark card.

**Avoids:** P2 (extraction failures), P6 (SPA injection timing), P11 (LLM unavailable states), P14 (tag inconsistency).

**Research flag:** Content extraction edge cases (SPAs, paywalls) warrant a short spike during phase planning to validate Readability.js behavior on a sample of real URLs.

---

### Phase 3: Search + Dashboard — Full Library Experience

**Rationale:** Once the AI pipeline produces reliable summaries, tags, and entities, search becomes meaningful. Dashboard browse and NL search can now be built on stable data.

**Delivers:** Full-page dashboard with bookmark grid/list, date/tag filters, natural language search in popup and dashboard (LLM query expansion over full-text), and settings (API key, provider selection).

**Addresses (features):** Dashboard browse view, tag and date filtering, NL search in popup, delete/archive, settings management.

**Avoids:** P7 (over-engineering search with vectors — ship LLM query expansion first), P13 (popup state lost on close), P15 (zero results for paraphrased queries).

**Research flag:** None — full-text search (Dexie) + LLM query expansion is a documented pattern.

---

### Phase 4: AI-Synthesized Answers — RAG over Bookmark Library

**Rationale:** "Ask your bookmarks a question and get a synthesized answer" is the headline differentiator. It requires stable content pipeline (Phase 2) and search (Phase 3) as foundations. This is the "second brain" moment.

**Delivers:** Dashboard search returns an AI-generated answer citing specific bookmarks, with source cards below. This is RAG over the user's personal library.

**Addresses (features):** AI-synthesized answers with source cards.

**Research flag:** RAG prompt design for bookmark retrieval is worth a targeted research spike during phase planning — especially context window management and citation formatting.

---

### Phase 5: Import Pipeline — Retroactive Bookmark Processing

**Rationale:** Import reuses the Phase 2 summarization pipeline code path with throttling and batch management added. Naturally last because it depends on a stable, production-tested pipeline and requires significant UX attention (progress, cost estimate, pause/resume).

**Delivers:** Users can import their existing Chrome bookmark library and get it summarized, tagged, and entity-extracted — with a cost estimate upfront, throttled processing, and a progress UI.

**Addresses (features):** Bookmark import, retroactive processing, data portability.

**Avoids:** P5 (API cost spike — cost estimate, throttled queue, idempotency).

**Research flag:** None — the pattern is well-defined in PITFALLS.md and ARCHITECTURE.md.

---

### Phase Ordering Rationale

- **Phase 1 before anything:** The alarm-driven queue is not optional and cannot be added later without restructuring core flows. Build it from the start.
- **Phase 2 before Phase 3:** Search over empty/bad summaries is useless. Pipeline quality must be validated before building search on top.
- **Phase 3 before Phase 4:** Synthesized answers require the search retrieval layer to exist.
- **Phase 5 last:** Import is a multiplier on Phase 2. Building it last means the pipeline it depends on has been validated at scale.
- **Entity extraction wired in Phase 2:** Low incremental cost during summarization; not surfaced in UI until Phase 4+, but data model must support it from the start for v2 knowledge graph.

### Research Flags

**Phases needing deeper research during planning:**
- **Phase 2 (content extraction):** Validate Readability.js against real-world URL samples (SPAs, paywalls, social media pages) before committing to the extraction architecture.
- **Phase 4 (RAG prompt design):** Research optimal RAG patterns for personal library search — context window budget, citation prompting, relevance ranking of retrieved bookmarks.

**Phases with standard, well-documented patterns (skip research-phase):**
- **Phase 1:** MV3 + Vite + @crxjs setup is extensively documented in the community.
- **Phase 3:** Dexie full-text queries + LLM query expansion are established patterns.
- **Phase 5:** Import throttling and cost estimation are covered directly in PITFALLS.md.

### Gaps to Address

- **Linear cosine similarity at scale (10K+ bookmarks):** Benchmarks suggest <50ms at 10K bookmarks, but this should be profiled during Phase 4 when embeddings are considered. If user corpora trend larger, the v2 HNSW path via `usearch` (WebAssembly) should be evaluated.
- **Readability.js on modern SPAs:** Extraction quality on JS-heavy pages (Twitter, YouTube, paywalled news) needs hands-on validation during Phase 2 planning. The fallback hierarchy is defined but real-world hit rates are unknown.
- **LLM output quality for summarization:** Tag consistency and summary quality are dependent on prompt design. This is not researchable in advance — it requires iteration during Phase 2 implementation. Budget for prompt engineering cycles.
- **v2 sync architecture:** Local-first is the right v1 choice, but the storage design should not make cloud sync impossible. Dexie's schema should include `updatedAt` and `deviceId` fields from Phase 1 to avoid a painful migration later.
